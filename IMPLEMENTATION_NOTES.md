# Remote Control Implementation Notes

## Scope

- Build a daemon-only uv-agent remote-control plugin with a modern web UI.
- Match the zcode remote-control interaction shape while adapting it to uv-agent:
  - Mobile first: workspace/task list -> thread session -> auxiliary panel.
  - Desktop: left project/thread sidebar, central thread workspace, optional right sidecar.
- Core work comes first: attachment-aware submit APIs, blob upload from plugins, canonical persistence, and tests.

## Confirmed Decisions

- The plugin runs only in daemon/persistent hosts.
- Default auth mode is `auth-code`; only explicit `auth.mode = "none"` disables auth.
- Realtime starts with SSE. Events are numbered incrementally and support replay via `Last-Event-ID`.
- The web client keeps a bounded local cache with eviction.
- Message send uses one multipart request containing JSON metadata and file parts.
- Attachments are uploaded only when the user clicks send.
- v1 supports image and general file attachments.
- Images keep TUI-style tokens such as `[Image #1]`; core validates the token and adds the image to model context.
- General files use visible tokens generated by the web UI. Core appends `id=<blob_id>` inside the token, persists that canonical text, and does not auto-read or inject file contents.
- User-typed token-looking text has no attachment meaning unless the web UI also sends matching structured attachment metadata.
- If a user wants literal `[Image #1]`, the web UI expects `" [Image #1]"` not to be used; the accepted literal form is `"[Image #1]"` or `“[Image #1]”` with no spaces between quotes and token.
- Attachment deletion is linked: deleting the chip/token removes the corresponding structured attachment before submit.
- Web upload limits: max 10 attachments, 50 MB per file, 100 MB per message by default, configurable.

## Core Attachment Contract

- Add a plugin-facing blob API, e.g. `context.blobs.put_bytes(...)`.
- Add `submit_turn(..., attachments=...)` through `PluginContext`, `AgentEngine`, and `TurnManager`.
- Core validates:
  - referenced blob exists;
  - attachment token exists exactly as expected in the submitted text;
  - image mime types are supported;
  - file tokens are bracketed so core can append `id=<blob_id>`;
  - invalid input fails before creating partial turn state.
- Core persists the canonical user input because future turns, compaction, replay, and audits must reconstruct the same model input.
